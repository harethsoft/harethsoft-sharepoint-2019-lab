**SharePoint Software Load Balancing via DNS Configuration**

 configuration of multiple Web Front End (WFE) servers and load balancing, here is the step-by-step guide for managing traffic distribution,


**Step 1: Design the Multi-Server Farm Architecture**
Establish a robust farm structure that includes a primary site with a database server, an application server, and **two Web Front End (WFE) servers** to handle initial client requests. 

**Step 2: Implement a Fault-Tolerant Secondary Site**
Configure a **secondary site** with its own set of WFE servers (e.g., wf3 and wf4) designed to take over the entire load as soon as the primary site fails.

**Step 3: Manage Logical vs. Physical Segregation**
Understand that while servers are logically segregated into primary and secondary sites, they **physically reside within one single farm** and are managed together in the SharePoint Central Administration.

**Step 4: Assign Dedicated Server Roles**
For ideal performance, ensure that every server has a **defined role**; the web application server, search service, application server, and distributed cache should ideally be separated onto different machines.

**Step 5: Identify the Web Application URL**
Determine the specific **prefix or URL** (e.g., hltftechcompany.com) that users will use within the domain to access the SharePoint site data.

**Step 6: Access the DNS Management Console**
On your domain controller, use the shortcut **"dnsmgmt.msc"** to access the DNS Management console where you will configure the software load balancer.

**Step 7: Create Host (A) Records for WFE Servers**
In the **Forward Lookup Zone**, add a new host entry for your site URL and define it against the **IP address of your first WFE server** (wf1).

**Step 8: Configure Multiple Entries for One Name**
Repeat the entry process for the **same site name** but define it against the **IP address of the second WFE server** (wf2), creating multiple records for a single URL.

**Step 9: Leverage DNS Round Robin Load Balancing**
Utilize the "beauty of Microsoft DNS" to **evenly divide incoming requests**; if 1,000 requests arrive, the DNS will automatically direct 500 to wf1 and 500 to wf2.

**Step 10: Execute Failover and Scaling Procedures**
If the primary site fails, **delete the primary DNS entries** and replace them with the IP addresses of the secondary WFE servers (wf3 and wf4). You can also **scale the farm** by simply adding more WFE IPs to the DNS list as load increases.

SHAREPOINT HIGH AVAILABILITY ARCHITECTURE
PRIMARY SITE AND SECONDARY SITE DESIGN

---

1. OVERVIEW OF THE ARCHITECTURE

This architecture is designed to provide:

High Availability (HA)
Disaster Recovery (DR)

The environment is divided into two locations:

Primary Site
Secondary Site

The Primary Site handles all user traffic under normal conditions.

The Secondary Site acts as a disaster recovery location and becomes active if the primary site fails.

The architecture contains three main layers:

1. Web Layer (WFE Servers)
2. Application Layer (Application Servers)
3. Database Layer (SQL Servers)

---

2. PRIMARY SITE COMPONENTS

The Primary Site is the main production environment.

It contains the following servers:

WFE-1
WFE-2
Application Server (Primary)
SQL Server (Primary Database)

---

2.1 WEB FRONT END SERVERS (WFE)

WFE Servers are responsible for:

Handling user requests
Rendering SharePoint pages
Hosting IIS web applications
Communicating with application services
Sending queries to SQL Server

Users never connect directly to the Application Server or SQL Server.

Instead, all user traffic goes through the WFE servers.

To improve performance and reliability, multiple WFE servers are used.

In this architecture:

WFE-1
WFE-2

---

2.2 APPLICATION SERVER

The Application Server runs SharePoint service applications and background processes.

Examples include:

Search Service
Managed Metadata Service
User Profile Service
Workflow Services
Excel Services
Business Connectivity Services

The Application Server performs processing tasks that should not run on WFE servers.

In this design:

App Server (Primary)

---

2.3 SQL SERVER (PRIMARY DATABASE)

SQL Server stores all SharePoint data.

Important databases include:

Configuration Database
Content Databases
Service Application Databases
Search Databases

The Primary SQL Server handles all read and write operations.

---

3. SECONDARY SITE COMPONENTS

The Secondary Site is used for Disaster Recovery.

It contains standby servers that mirror the Primary environment.

Servers in the Secondary Site:

WFE-3
WFE-4
Application Server (Secondary)
SQL Server (Secondary)

These servers remain idle or partially active until a failover occurs.

---

4. DATABASE REPLICATION

The database layer uses SQL Server replication technologies to synchronize data between sites.

The most common solution is:

SQL Server Always On Availability Groups

In this configuration:

Primary SQL Server = Read and Write
Secondary SQL Server = Replica

The Primary SQL continuously replicates data to the Secondary SQL server.

Two replication modes exist:

Synchronous replication
Asynchronous replication

Synchronous replication provides zero data loss but requires low latency between servers.

---

5. SHAREPOINT FARM CONNECTION TO SQL

All SharePoint servers (WFE and Application) connect to SQL Server.

Under normal conditions, they connect to:

Primary SQL Server

If the primary database fails, a failover occurs and SharePoint servers reconnect to:

Secondary SQL Server

This process is usually managed by SQL Availability Group Listener.

---

6. LOAD BALANCER CONFIGURATION WITH WFE SERVERS

The Load Balancer distributes user traffic across multiple WFE servers.

Without a Load Balancer, users would connect to only one WFE server, which creates a single point of failure.

---

6.1 DNS CONFIGURATION

Users access SharePoint using a URL such as:

[https://portal.company.com](https://portal.company.com)

DNS does NOT point directly to a WFE server.

Instead, DNS points to the Load Balancer IP address.

Example:

portal.company.com → Load Balancer IP

---

6.2 LOAD BALANCER TRAFFIC FLOW

User Request Flow:

User Browser
↓
DNS Resolution
↓
Load Balancer
↓
WFE Server (WFE-1 or WFE-2)
↓
Application Server
↓
SQL Server

The Load Balancer distributes incoming requests between WFE-1 and WFE-2.

---

6.3 LOAD BALANCING METHODS

Common algorithms used by Load Balancers include:

Round Robin
Least Connections
Weighted Distribution

Example:

User 1 → WFE-1
User 2 → WFE-2
User 3 → WFE-1

This balances server workload.

---

6.4 HEALTH CHECKS

The Load Balancer performs health checks to verify that WFE servers are operational.

Example health check:

HTTP request to

[http://WFE1/_layouts/15/healthcheck.aspx](http://WFE1/_layouts/15/healthcheck.aspx)

If a server fails the health check, the Load Balancer automatically removes it from the pool.

---

7. FAILOVER SCENARIO

If the Primary Site becomes unavailable:

Step 1
SQL failover occurs and the Secondary SQL Server becomes active.

Step 2
DNS is updated to redirect traffic to the Secondary Site.

Step 3
Load Balancer in the Secondary Site directs traffic to:

WFE-3
WFE-4

Step 4
Application Server (Secondary) begins handling service application workloads.

Users continue accessing SharePoint with minimal interruption.

The **Load Balancer** is not only one specific thing. It can be a **hardware device, software application, or cloud service**, depending on the infrastructure design.

In simple terms:

Load Balancer = A system that distributes user requests across multiple servers to prevent overload and improve availability.

---

1. LOAD BALANCER AS HARDWARE

In large enterprise environments, the Load Balancer is usually a **dedicated hardware appliance**.

Common examples include:

* F5 Networks BIG-IP
* Citrix NetScaler
* Fortinet FortiADC

These devices are installed in the network **in front of the web servers**.

Traffic flow example:

User
↓
DNS
↓
Load Balancer Device
↓
WFE-1
WFE-2

Advantages:

* Very high performance
* Handles thousands of users
* Advanced health monitoring
* SSL offloading capabilities

---

2. LOAD BALANCER AS SOFTWARE

A Load Balancer can also be **software running on a server**.

Examples include:

* NGINX
* HAProxy
* Apache HTTP Server (Reverse Proxy)

In this case, the load balancing service runs on a Linux or Windows server.

Traffic flow example:

User
↓
DNS
↓
Load Balancer Server (for example NGINX)
↓
WFE-1
WFE-2

---

3. LOAD BALANCER AS A CLOUD SERVICE

In cloud environments, the Load Balancer is usually provided as a **managed cloud service**.

Examples:

* Microsoft Azure Load Balancer
* Amazon AWS Elastic Load Balancer

These services automatically distribute traffic between virtual machines.

---

4. LOAD BALANCER USING WINDOWS (NLB)

In Microsoft environments, you can also use a built-in Windows feature called:

Network Load Balancing (NLB)

This feature is included in Windows Server and allows multiple servers to share a single virtual IP address.

Example:

WFE-1 → 10.0.0.10
WFE-2 → 10.0.0.11

Cluster Virtual IP:

10.0.0.20

DNS record:

portal.company.com → 10.0.0.20

Requests sent to the virtual IP are automatically distributed between the WFE servers.

---

5. HOW LOAD BALANCER IS USED WITH SHAREPOINT

In SharePoint On-Premises environments, the Load Balancer is placed **in front of the Web Front End (WFE) servers**.

Architecture example:

Users
↓
DNS
↓
Load Balancer
↓
WFE-1
WFE-2
↓
Application Server
↓
SQL Server

Purpose of using a Load Balancer:

* Distribute user traffic across servers
* Improve system performance
* Prevent service interruption if one WFE server fails
* Provide high availability

---

SUMMARY

A Load Balancer can be implemented as:

Hardware appliance
Software application
Cloud service
Windows Server feature (NLB)

Regardless of the implementation, the main function of a Load Balancer is:

Distributing incoming user requests across multiple servers to ensure performance, scalability, and high availability.
